# 2026-06-01 学习日志

## 知识点

- [AI 工作流] cbc -p 模式调用项目级 slash command 的真实行为 → 详见 [topics/ai/workflow/](/topics/ai/workflow/)
  - cbc -p **加载**项目级 `.codebuddy/commands/<namespace>/*.md` 但**不展开**模板
  - 复用 SOP 命令唯一路径：主程序自己读 .md 全文拼到 prompt 头

## 问题与解决

- **问题**: 给 v0.3 orchestrator 设计 apply / e2e cbc 调用方式时，本想直接 `cbc -p "/opsx:apply ..."` 复用 infrastructure-web 已有的 14 个 opsx SOP 命令，避免另写 5 个 skill 的提示词工程成本
- **解决**: 通过三次实验确认 cbc -p 不展开 slash 命令模板（只加载到 init 列表里供识别但不执行模板），改成"主程序读 commands/opsx/xxx.md 全文拼到 prompt 头"方案 E。`CodeBuddyClient.run()` 加 `command_file` 参数，apply.py / e2e.py 改引用 commands/opsx/{apply,e2e,cr}.md

## 经验教训

- **不准猜 + 实验交叉验证**: 第一次实验 `cbc -p "/opsx:explore ..."` 时 init 列表没出现 opsx，得出"cbc -p 不加载项目级命令"的结论。第二次冷启动后 cwd 缓存建好，init 列表含 14 条 opsx——**首次冷缓存可能假阴性**，必须复测才下结论
- **细分"加载" vs "执行"**: 一开始把"slash 命令出现在 init 列表"等同于"cbc 会执行模板"——实际上 IDE 交互模式才展开模板，-p 非交互模式只注册不执行
- **诚实纠偏**: 之前误以为"v0.2.0 也没这个 skill"是历史遗留，git show v0.2.0 验证后发现 v0.2.0 时 orchestrator/ 目录根本不存在——5 个 skill 是 v0.3 T2.6 commit `48c8b71` 主理人凭印象 hardcode 的，挂着 TODO 一直没回头补

副线：通过三次实验定型 v0.3.1 真链路打通方案 E——复用 v0.2 已沉淀的 opsx SOP，主程序拼 prompt，避免重写 5 个 skill。已派 worker 实施。
